## چک‌لیست سریع
- [ ] آیا پروژه اوپن‌سورس و مرتبط با ضدجعل/ضدفرا‌د است؟
- [ ] فیلدهای الزامی (`name`, `github`, `url`, `category`, `description`, `license`) را در `projects.json` پر کرده‌اید؟
- [ ] توضیح فارسی کوتاه (۱–۲ خط) و بی‌طرف نوشته‌اید؟
- [ ] `category` یکی از مقادیر مجاز است و `tags`/`tech` در صورت وجود آرایه هستند؟
- [ ] اسکریپت‌های اعتبارسنجی را اجرا کرده‌اید؟
- [ ] اگر سناریو یا Playbook اضافه کرده‌اید، لینک و ساختار خوانا در `docs/` دارد؟

## این فهرست برای چه کسانی است؟
- مهندسان امنیت و تیم‌های محصول که می‌خواهند ضدجعل را در MVP یا محصول نهایی تعبیه کنند.
- پژوهشگران دیپ‌فیک، فورنزیک رسانه، هویت دیجیتال و امنیت زنجیره تأمین.
- جامعه متن‌باز که به دنبال مرجع مرتب برای اصالت‌سنجی داده و هویت است.

## انواع مشارکت
- **افزودن پروژه‌های اوپن‌سورس مرتبط**: با تکمیل `projects.json` و اجرای اسکریپت‌های اعتبارسنجی.
- **بهبود سناریوها و استک‌ها**: افزودن یا به‌روزرسانی سناریوهای عملی در `docs/use-cases-and-stacks.fa.md`.
- **نوشتن مثال‌های عملی یا Playbook**: آموزش‌های کوتاه و Playbookهای استقرار و ممیزی در پوشه `docs/` با ساختار مشخص.

## چه پروژه‌هایی پذیرفته می‌شوند؟
- سامانه‌های اعتبارسنجی اسناد و گواهی‌ها، OpenBadges، Blockcerts.
- پلتفرم‌های IAM، SSO، OAuth2/OIDC، MFA، DID و verifiable credentials.
- ابزارهای زنجیره تأمین (فیزیکی یا نرم‌افزار)، provenance، SBOM و DKG.
- ابزارهای تشخیص دیپ‌فیک، فورنزیک رسانه و دیتاست‌های مرتبط.
- کتابخانه‌های رمزنگاری و امنیتی که مستقیم برای امضا، هش و اصالت داده کاربرد دارند.

### چه پروژه‌هایی رد می‌شوند؟
- ابزارهای دیپ‌فیک/جعل صوت و تصویر برای استفاده غیراخلاقی یا بدون رضایت.
- پروژه‌های misinformation و انتشار محتوای ساختگی.
- ریپوهای بدون لایسنس مشخص، خصوصی یا فاقد ارتباط روشن با ضدجعل.

## چه مشارکت‌هایی بیشترین ارزش را در موج پنج ایجاد می‌کنند؟
- سناریوهای واقعی و مستند از پیاده‌سازی ضدجعل در سازمان‌ها یا محصولات.
- توضیح و کاهش ریسک‌های اخلاقی و حاکمیتی در استفاده از ابزارها و مدل‌های تشخیص.
- بهبود اسکریپت‌ها و اتوماسیون برای اعتبارسنجی داده‌ها و کاهش خطای انسانی در جمع‌آوری لیست.

## ساختار JSON (الزامی‌ها)
فایل [docs/SCHEMA.md](docs/SCHEMA.md) را برای توضیح کامل ببینید. فیلدهای زیر **باید** پر شوند:
- `name`
- `github`
- `url`
- `category` (یکی از: `certificates`, `identity`, `supply_chain`, `media_forensics`, `library`)
- `description` (فارسی، ۱–۲ خط)
- `license`

فیلدهای پیشنهادی: `tech`, `tags` (حروف کوچک، جداشده با `-` در صورت نیاز؛ مثال: `supply-chain`, `deepfake`, `signature`).

## قوانین نگارشی
- توضیح پروژه را فارسی، خلاصه و بی‌طرف بنویسید؛ از لحن تبلیغاتی پرهیز کنید.
- `tags` را با حروف کوچک بنویسید و از فاصله استفاده نکنید (مثال: `deepfake-detection`).
- `github` می‌تواند `owner/repo` یا فقط نام سازمان باشد؛ مقدار خالی پذیرفته نیست.

## مراحل ارسال Pull Request
1. ریپو را Fork و شاخهٔ جداگانه بسازید.
2. ورودی جدید را به `projects.json` اضافه کنید و فیلدهای الزامی را پر کنید.
3. اسکریپت‌ها را اجرا کنید:
   ```bash
   python -m json.tool projects.json
   python scripts/validate_projects.py
   python scripts/generate_readme.py
   ```
4. README به‌صورت خودکار بین دو نشانگر تولید می‌شود؛ از ویرایش دستی جدول‌ها خودداری کنید.
5. توضیح مختصر درباره تغییرات در PR بنویسید و اگر لینک یا لایسنس نامشخص است، در توضیحات PR اشاره کنید.

## چک‌لیست Pull Request
- [ ] اگر پروژه جدید اضافه کرده‌ام، هم در README و هم در `projects.json` ثبت شده است.
- [ ] دسته‌بندی (`category`) درست و یکی از مقادیر مجاز است.
- [ ] توضیح فارسی پروژه کوتاه، شفاف و بدون اغراق است.
- [ ] اگر سناریو یا Playbook جدید اضافه کرده‌ام، در docs لینک و ساختار قابل‌خواندن دارد.
